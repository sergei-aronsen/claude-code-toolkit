🧭 Product Thinking customize

Validation gate skill: forces you to answer who/why/metric/channel/economics BEFORE writing code. Auto-triggers on "build/MVP/ship/pricing/launch/pivot" prompts.

1) Customize your domain config

The skill reads ~/.claude/product-config.json for ICP-specific gates. Without it, generic gates apply (B2B floor $1000/yr, LTV/CAC ≥ 3). With it, gates match your real market.

Copy the template and edit:

cp ~/.claude/skills/product-thinking/config-template.json ~/.claude/product-config.json
$EDITOR ~/.claude/product-config.json

Key fields:

  • target_market — e.g. "Nordic SMB", "US SMB", "EU SMB", "LatAm SMB", "Global"
  • b2b_price_floor_usd_yearly — if a B2B feature prices below this, the skill blocks with a "SaaS graveyard" pushback. Default 1000.
  • b2c_price_ceiling_usd_monthly — high-friction gate for B2C. Default 100.
  • ltv_cac_minimum — economics gate. Default 3.0.
  • structural_advantages — list of structural wedges (BankID, EU residency, local language, etc.) that the skill suggests during the competitor question.

2) Tune the pushback rules

The skill ships with universal pushbacks ("marketing later", "no competitors", "MVP without metric", etc.). You can edit them inside ~/.claude/skills/product-thinking/SKILL.md at the section "Pushback Rules". Add your own based on past mistakes.

Common edits:

  • Add a row for your specific anti-pattern (e.g. "let's use RAG" → "RAG isn't required for MVP, ship simpler").
  • Soften the SaaS graveyard gate if you sell to a niche where $300/yr is normal (e.g. consumer prosumer).
  • Tighten the channel pushback list — add platforms you've personally seen fail.

3) Tune the SaaS graveyard gate

The hardest gate is the B2B SMB pricing floor. Two ways to tune:

  • Adjust threshold in ~/.claude/product-config.json → b2b_price_floor_usd_yearly. Lower for vertical niches, raise for enterprise.
  • Disable per-feature by adding an exception to the feature's .planning/product/<slug>.md with status: risk-accepted and a written acknowledgment of the unit-economics risk.

4) UserPromptSubmit hook (optional but recommended)

The skill ships with a soft reminder hook at .claude/hooks/product-gate.sh. When you type a feature-related prompt, it injects a one-line nudge to run product-thinking first.

Disable globally:

export CLAUDE_PRODUCT_GATE=0

Skip per-prompt: add (no-product-gate) to your message — the hook detects it and stays silent.

Make it stricter (block instead of suggest): route the same script through PreToolUse on Edit|Write and exit non-zero when .planning/product/<slug>.md is missing for the current feature.

5) Daily commands

  • /product-review "feature description" — runs 4 personas (skeptic + marketer + CFO + user-empath) on a draft.
  • /product-review --lite "small change" — 3-question fast review for trivial work.
  • /product-review --persona cfo "..." — only one persona (skeptic / marketer / cfo / user-empath).
  • /product-review --council "..." — combined business + technical review (uses the Supreme Council infrastructure).

6) Where outputs land

Validation gate output: .planning/product/<feature-slug>.md

Multi-persona review output: .planning/product/review-<slug>-YYYY-MM-DD.md

Both files become input context for /gsd-discuss-phase and /gsd-plan-phase — your business answers persist into technical planning.

Hidden risks

  • Validation theater — filling the markdown gives false confidence. Real validation = paying customer. The file is hypothesis, not proof.
  • Prompt fatigue — after 2 weeks you may resent the gate. Use --lite for trivial work; the skill auto-skips bug fixes / refactors / docs / lint via anti-trigger keywords.
  • Bypass abuse — if every feature ends up risk-accepted, the gate becomes noise. Track frequency manually or add a probe to /vendor-audit.
