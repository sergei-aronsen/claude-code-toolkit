## Implementation Decisions

### Translation scope — full rewrite

- **D-01:** Rewrite each translation from the current English `README.md` top-to-bottom.
  Do NOT patch-translate only the changed sections.
- **Why:** Phase 6 reshuffled the whole README (Install Modes block is new, MCP table changed,
  Components section was added, Killer Features cell rewritten). Section-by-section diff +
  patch leaves stale v3.x paragraphs interleaved with v4.0 prose — impossible to audit.
- **Log:** `[auto] Translation scope — Selected: "Full rewrite" (recommended — prevents v3.x/v4.0 drift inside one file)`

### Execution strategy — grouped parallel waves

- **D-02:** 3 plans, 2–3 languages each, executable in parallel.
  - **Plan 06.1-01:** European Romance + Germanic — `es`, `pt`, `fr`, `de` (4 langs)
  - **Plan 06.1-02:** East Asian CJK — `zh`, `ja`, `ko` (3 langs)
  - **Plan 06.1-03:** Cyrillic — `ru` (1 lang)
- **Why:** One plan per language = 8 waves of commit churn, diluted review. One mega-plan =
  impossible atomic rollback if a single language breaks mdlint. Language-family grouping
  keeps related typography/punctuation rules together (e.g., CJK width, Romance accents,
  Cyrillic) and caps blast radius per plan. 06.1-03 isolates `ru` because the project
  maintainer is a native speaker — that plan gets the tightest scrutiny.
- **Log:** `[auto] Execution strategy — Selected: "Grouped family waves (3 plans)" (recommended — balance parallelism, review surface, native-speaker isolation for ru)`

### Translation source — Claude-produced inline

- **D-03:** Claude produces translations during plan execution. No external translator
  service, no per-language user review gate inside the chain.
- **Why:** `--chain` mode runs plan → execute autonomously. A human-in-the-loop gate per
  language breaks the chain. Claude translation quality is adequate for a project README
  (domain-specific tech terms dominate). Native-speaker polish is a deferred v4.1 task.
- **Log:** `[auto] Translation source — Selected: "Claude inline during execute" (recommended — compatible with --chain autonomous flow)`

### Technical term handling — code English, prose localized

- **D-04:** Keep unchanged in English: code blocks, shell commands, CLI flags, file paths,
  package names, env var names, badge alt text, URL anchors.
- **D-05:** Translate: section headings, prose paragraphs, table cell descriptions, table
  column headers, admonition labels (Note/Warning/Security).
- **D-06:** Preserve as-is (proper nouns): `Claude Code`, `superpowers`, `get-shit-done`,
  `RTK`, `MCP`, `Supreme Council`, `Gemini`, `ChatGPT`, framework names (`Laravel`, `Rails`,
  `Next.js`, `Node.js`, `Python`, `Go`), server names (`context7`, `playwright`,
  `sequential-thinking`, `sentry`, `dbhub`).
- **Why:** Proper nouns break searchability if translated. Code must be copy-pasteable
  verbatim. Headings/prose in target language preserve reading flow.
- **Log:** `[auto] Technical terms — Selected: "Code English, headings + prose localized, proper nouns preserved" (recommended — standard localization pattern)`

### Howto link targets — point to translated howto

- **D-07:** Each translation's "step-by-step installation guide" link points to
  `../howto/{lang}.md` (e.g., `ru.md` → `../howto/ru.md`). Fallback to `en.md` only if the
  localized howto is missing.
- **Why:** `docs/howto/` already has all 9 languages (`en, de, es, fr, ja, ko, pt, ru, zh`).
  The existing `ru.md` translation already follows this pattern (`docs/readme/ru.md:10`
  → `../howto/ru.md`). Keep consistent.
- **Log:** `[auto] Howto link target — Selected: "Link to translated howto, en fallback only if missing" (recommended — all 9 howto languages exist; zero fallback needed)`

### Language switcher row — preserve structure, highlight current

- **D-08:** Keep the same 9-item language list order as `README.md:8`. Current language =
  plain text (no link); all others = link to sibling `*.md`. English link points to
  `../../README.md` from a translation file.
- **Why:** This mirrors existing `docs/readme/ru.md:8` pattern and keeps switcher visually
  identical across all languages (helps users orient).
- **Log:** `[auto] Language switcher — Selected: "Same order, current as plain text, sibling paths" (recommended — matches existing ru.md pattern)`

### Line-count targeting — mirror English structure 1:1

- **D-09:** Mirror `README.md` section-by-section with equivalent paragraph count. Natural
  language expansion (German +15–20%, French/Spanish/Portuguese +10–15%, CJK typically
  within −5% to +5%, Russian +5–10%) keeps every translation comfortably inside the
  161–242 line band without padding or truncation.
- **D-10:** If a translation lands below 161 or above 242, rebalance paragraphs (merge or
  split) rather than padding with filler or deleting content.
- **Why:** Content fidelity first, band compliance second. The ±20% band exists to catch
  stale translations, not to force artificial length.
- **Log:** `[auto] Line-count target — Selected: "Mirror English 1:1, natural expansion handles band, rebalance if out" (recommended — content parity beats length gaming)`

### Required Base Plugins mention — prose localized, commands English

- **D-11:** When rendering the Install Modes / complement positioning prose, translate the
  surrounding explanation but keep the exact install commands verbatim:

  ```bash
  claude plugin install superpowers@claude-plugins-official
  bash <(curl -sSL https://raw.githubusercontent.com/gsd-build/get-shit-done/main/scripts/install.sh)
  bash <(curl -sSL .../scripts/migrate-to-complement.sh)
  ```

- **Why:** Commands are copy-paste surface — they must match the English source exactly
  so users can run them without edits. Prose around them explains intent in the user's
  language.
- **Log:** `[auto] Base plugins block — Selected: "Localize prose, keep commands verbatim" (recommended — copy-paste integrity)`

### Quality validation — mdlint + drift + visual diff

- **D-12:** Per-translation checklist:
  1. `npx markdownlint-cli "docs/readme/{lang}.md"` clean
  2. `make translation-drift` passes after all 8 written
  3. Visual side-by-side section count vs `README.md` (same section headings present)
  4. Spot-check: language switcher row has correct current-language plain text, all 8
     siblings as links, English points to `../../README.md`
  5. Spot-check: all code blocks verbatim-equal to `README.md`
- **D-13:** Defer: native-speaker QA pass across all 8 languages (v4.1 backlog).
- **Why:** mdlint + drift are machine-enforceable gates; visual/spot checks are cheap for
  a single reviewer; native QA is expensive and not required for v4.0 ship.
- **Log:** `[auto] Quality validation — Selected: "mdlint + drift + 3 spot-checks; defer native QA to v4.1" (recommended — ships v4.0 without adding human-review dependency)`

### Commit granularity — one commit per language

- **D-14:** Each of the 8 translated files ships in its own commit within its plan's atomic
  group. Conventional commit shape: `docs(06.1): sync {lang} README to v4.0 complement-first`.
- **Why:** Per-language commits let git blame surface translation history cleanly and
  allow per-language revert if a specific translation needs rework.
- **Log:** `[auto] Commit granularity — Selected: "One commit per language inside plan wave" (recommended — clean history, revertible)`

### Markdownlint MD024 siblings — heading deduplication

- **D-15:** Translations are free to produce heading text identical to other translations
  (same English source → same translated heading pattern is expected). `.markdownlint.json`
  already sets `MD024.siblings_only: true`, so duplicate headings across sibling docs do
  not trip lint. Within a single translation, each `##`/`###` must remain unique within
  its parent — same constraint as English `README.md`.
- **Why:** Pre-empt a foreseeable lint failure mode; confirms existing config covers the
  cross-file case.
- **Log:** `[auto] Lint config verification — Selected: "Rely on existing MD024.siblings_only=true" (recommended — no config change needed)`

### Claude's Discretion

- Exact translated phrasing, idiomatic choices, punctuation preferences (e.g., em-dash
  vs. en-dash vs. hyphen usage per language) — implementer picks per-language conventions
  as long as mdlint passes.
- Ordering of sub-items within tables (e.g., alphabetical vs. frequency) — keep English
  source order unless a language-specific sort is natural (e.g., CJK collation for server
  names — not applicable here because server names stay English).
- Whether to use formal ("vous", "Sie", "вы", "usted") or informal register — default to
  formal for all languages (README audience = developers, formal is the professional norm
  and matches existing `ru.md` tone).

### Translation register per language

All 8 use formal register. Key conventions:

- **de:** `Sie` form, `Installation` rather than `Installieren`, preserve English technical
  nouns (`Hook`, `Plugin`, `Skill`, `Template`) as German nouns with capitalization.
- **es / pt:** formal `usted` / `você`, preserve tech nouns as cognates where natural
  (`plugin` — plugin, `hook` — hook, `agente` — agent).
- **fr:** `vous` form, `plugin` as masculine, avoid excessive Anglicisms (`configuration`
  not `setup`).
- **ja:** です/ます form, preserve English tech terms in katakana or Latin as convention
  dictates (`プラグイン`, `フック`, `コマンド`).
- **ko:** `-습니다` form, tech terms preserved in Hangul phonetic or Latin.
- **zh:** Simplified Chinese (zh-Hans), formal written register, preserve CLI
  commands/flags in English.
- **ru:** formal `Вы`, preserve tech nouns as transliterated or English (хук/hook,
  плагин/plugin — use whichever the existing `ru.md` already established for consistency).

### What NOT to do

- Do NOT translate or rename section anchors mid-flight — breaks in-README cross-links.
- Do NOT translate command output examples inside fenced code blocks — users grep for
  exact strings.
- Do NOT skip the `Required Base Plugins` implicit mention — every translation must
  convey "TK is a complement, not a replacement" at least once in the Install Modes
  section.
