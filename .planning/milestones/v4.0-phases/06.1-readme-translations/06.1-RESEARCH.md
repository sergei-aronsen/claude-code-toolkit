## Domain Overview

Phase 06.1 is a pure documentation sync task: rewrite 8 Markdown files so each mirrors the
202-line English `README.md` that Phase 6 rewrote with complement-first positioning. No code
changes. No new Makefile targets. No manifest changes.

The mechanical gates are already wired:

- `make translation-drift` (Makefile line 178–197) computes `README_LINES=$(wc -l < README.md)`,
  then checks every `docs/readme/*.md` file for `LINES < MIN (80%)` or `LINES > MAX (120%)`.
- `make mdlint` runs `markdownlint '**/*.md' --ignore-path .markdownlintignore`.
- Both are prerequisites of `make check`, which blocks Phase 7 Plan 07-04.

The 8 translations are currently 148 lines each — all below the 161-line floor. They still
reflect v3.x: no Install Modes block, missing dbhub and sentry MCP entries, stale Killer
Features rows, no complement-first positioning, no "Required Base Plugins" framing.

The research below answers: how do we write each replacement file so it passes the gates and
reads naturally in each target language?

## Per-Language Conventions

### German (de)

**Register:** `Sie` (formal), verb second position in main clauses. The existing `de.md`
already uses informal `du`/`dein` in headings like "Fuer wen ist das" and "dein-projekt/".
The v4.0 rewrite should adopt formal `Sie` consistently to match the CONTEXT.md default
and to align with other translations (de.md v3.x was the one outlier with informal headings).

**Punctuation:**

- Em-dash: use ` — ` (spaced) or `–` (en-dash), either is common in German technical
  writing. Avoid bare `—` without spaces (non-standard in German typography).
- Quotation marks: German uses „lowered-99" and "raised-66" (`„..."`) but in markdown
  tech docs, `"..."` ASCII quotes are standard and pass lint cleanly. Use ASCII.
- No space before `:` or `!` (unlike French).
- Ellipsis: `...` (ASCII triple dot) is safe; `…` Unicode is fine but not required.

**Loanwords and tech nouns:**

- German capitalizes all nouns. Treat adopted English tech terms as German nouns:
  `der Hook`, `das Plugin`, `der Skill`, `das Template`, `der Agent`. Write with capital
  when used as nouns in prose.
- `Framework` → capitalize as `Framework` (German noun).
- `Toolkit` → `Toolkit` (no translation needed, widely used in German tech).
- `Slash-Befehl` or `Slash Command` — the existing de.md uses `Slash Commands` (English).
  Keep `Slash Commands` (English plural) in headings for searchability, or use `Slash-Befehle`
  in prose for readability. Either passes lint.
- `Workflow` → `Workflow` (no translation).

**Markdownlint gotchas for de:**

- Umlaut characters (ä, ö, ü, Ä, Ö, Ü, ß) in headings are safe — markdownlint operates on
  UTF-8 and does not flag non-ASCII in headings.
- MD026 (no trailing punctuation in headings): "Für wen ist das" — the question framing
  without `?` is valid German and safe. Do NOT add `?` to headings.
- The existing de.md uses `ae`, `oe`, `ue` fallbacks (e.g., "fuer", "fuer"). The v4.0
  rewrite MUST use proper Unicode (ä, ö, ü) to read naturally. These are safe in Markdown.

**Length factor:** German typically runs +15–25% longer than English in technical prose due to
compound nouns and inflected forms. See Length Expansion Profile below.

### Spanish (es)

**Register:** `usted` (formal second person singular), which makes verbs third person singular
(`instale`, `ejecute`, `configure` rather than `instala`, `ejecuta`, `configura`). If targeting
Latin America primarily, `usted` is the universal formal choice.

**Punctuation:**

- Inverted question/exclamation marks at sentence start (`¿`, `¡`) — use only in prose
  sentences that are questions or exclamations. Avoid in headings (MD026 safety — the inverted
  marks are not in markdownlint's trailing punctuation set, but it is safer to keep headings
  without them). In the README, most headings are declarative nouns — no issue.
- Em-dash: ` — ` (spaced) or `—` — both acceptable in Spanish technical writing.
- Ellipsis: `...` ASCII triple dot.
- Quotation marks: `«...»` (guillemets) or `"..."` ASCII. Use ASCII for markdown safety.

**Loanwords and tech nouns:**

- Spanish nouns are gendered. Common tech loanwords: `el plugin`, `el hook` (masculine),
  `la plantilla` (template — existing Spanish translations tend to use `plantilla` or
  keep `Template`). For this README keep `Template` English as it matches the UI.
- `Toolkit` → `toolkit` (lowercase when used generically in prose; keep `Toolkit` when
  part of proper name "Claude Code Toolkit").
- `Slash commands` → `comandos slash` or `slash commands`. Keep English for searchability.
- `Framework` → `framework` (masculine, no accent needed).
- Accented characters (á, é, í, ó, ú, ñ, ü) — safe in markdown headings.

**Markdownlint gotchas for es:**

- Accented characters in headings are fully supported.
- MD026 is safe for Spanish as long as headings are not questions (no trailing `?`).

**Length factor:** Spanish is +10–15% longer than English.

Use `você` — it is the dominant tech-doc register for Brazilian Portuguese, which is the larger

**Punctuation:**

- Em-dash: ` — ` (spaced). In Brazilian Portuguese, the em-dash is common in dialogue but
  less so in technical prose; a colon or parentheses may be more natural in some cases.
- Quotation marks: `"..."` ASCII is standard for Brazilian PT tech docs.
- No space before `:` (unlike French).
- Accented characters (á, â, ã, à, é, ê, í, ó, ô, õ, ú, ç) — safe in markdown.

**Loanwords and tech nouns:**

- Portuguese absorbs English tech terms more than Spanish: `plugin`, `hook`, `template`,
  `framework` all used as-is (lowercase, masculine).
- `Toolkit` → `toolkit` in prose, `Toolkit` in proper name.
- `Slash commands` → `comandos slash` or keep English.

**Markdownlint gotchas for pt:**

- Accented characters are fully supported.
- MD026 is safe for declarative headings.

**Length factor:** Brazilian Portuguese is +10–15% longer than English.

### French (fr)

**Register:** `vous` (formal), conjugate with `vous`. France developer audience tends to use
formal register in documentation.

**Punctuation (critical for French):**

- Space before `?` and `!` and `:` — this is mandatory in French typography: `Démarrer ?`,
  `Installer :`. However, this can theoretically trigger markdownlint MD031/MD032 inside
  table cells or list items where whitespace handling is aggressive. In practice,
  `markdownlint-cli` does NOT flag spaces before `:` in prose — only checks specific rules.
  Use the French typographic space (` :`, ` ?`, ` !`) in prose. Inside code spans or
  code blocks, keep English conventions verbatim (already handled by D-04).
- Guillemets for quotation: `«...»` with non-breaking thin spaces inside — complex in
  markdown. Use `"..."` ASCII quotes instead for lint safety.
- Em-dash: ` — ` (spaced). In French, the em-dash is used with spaces.
- Ellipsis: `...` or `…` — both safe.

**Loanwords and tech nouns:**

- French is protective of its language but tech docs commonly keep English terms.
  `plugin` (masculine, no accent), `hook` (masculine), `template` (masculine, or use
  `modèle` as alternative), `framework` (masculine).
- L'Académie française recommends `module d'extension` for plugin — irrelevant for a
  tech README; keep `plugin`.
- `Slash commands` → `commandes slash` or `commandes /`.
- `Toolkit` → `toolkit` in prose, `Toolkit` in proper name.
- Accented characters (é, è, ê, ë, à, â, ù, û, î, ï, ç, œ, æ) — safe in markdown.
- Proper adjective agreement: "le serveur `context7`" (masculine server name, kept as-is).

**Markdownlint gotchas for fr:**

- The French typographic space before `:` in prose (`Configuration :`) is safe in
  markdownlint. It does NOT trigger MD027 or MD010 (those apply to list markers and
  code indentation, not inline prose punctuation).
- MD026 (no trailing punctuation in headings): headings in French should be declarative
  nouns without trailing `:`. Do NOT add `:` at end of headings.
- Accented characters in headings are fully supported.

**Length factor:** French is +10–15% longer than English.

### Chinese Simplified (zh)

**Register:** Formal written Chinese. No pronoun inflection needed (Chinese is not heavily
inflected). Address the reader implicitly or use `您` (formal) when direct address is needed.

**Punctuation:**

- Chinese uses fullwidth punctuation: `，` (comma), `。` (period), `：` (colon), `！`
  (exclamation), `？` (question), `（` `）` (parentheses).
- For markdown, keep ASCII punctuation in inline code spans and code blocks (D-04).
- In prose headings, Chinese headings typically end without punctuation — safe for MD026.
- Quotation marks: `「...」` (Taiwan) or `"..."` fullwidth (Mainland). Use `"..."` (fullwidth
  or ASCII) for simplicity.
- No space between Chinese characters. Space only before/after Latin/ASCII segments
  (standard CJK-Latin boundary spacing).

**Loanwords and tech nouns:**

- Tech terms: keep English for searchability (`Claude Code`, `MCP`, `RTK`).
- Common Chinese tech vocabulary: `命令` (command), `模板` (template), `框架` (framework),
  `插件` (plugin), `钩子` or `Hook` (hook — keep English), `代理` (agent), `技能` (skill),
  `规则` (rules), `工作流` (workflow).
- Slash commands: `斜杠命令` (used in existing zh.md).
- Install: `安装` (existing zh.md uses this).
- Framework: `框架`.
- Standalone: `独立安装` (literal translation is natural).
- Complement: `补充安装` or `增强安装`.

**Markdownlint gotchas for zh:**

- CJK characters are multi-byte but markdownlint processes Unicode correctly.
- MD013 (line length) is disabled — irrelevant. CJK-dense lines can be very long in
  bytes but short visually.
- Table cell width: CJK characters are double-width in monospace terminals (CJK Ambiguous
  Width or CJK Full Width). This affects table alignment visually in terminal `cat` output
  but does NOT affect markdownlint table rules. GitHub renders correctly.
- MD040 (fenced code block language) — ensure all code blocks have a language specifier.
  The existing zh.md uses `bash` on all fences (confirmed from reading the file).
- The Structure After Installation `text` fence block contains Chinese comments like
  `# 主要指令`. The existing zh.md marks it as ` ```text ` which is correct.

**Length factor:** Chinese is typically −5% to +5% relative to English. Dense CJK prose
is more compact per character than English, but Chinese technical writing often adds
explanatory parentheticals. Net: expect zh to land close to 202 lines.

### Japanese (ja)

**Register:** `です/ます` form (polite register, standard for developer documentation).
Avoid `だ/である` (plain form — too casual/assertive) and ultra-honorific (`いらっしゃいます`).

**Punctuation:**

- Japanese uses `、` (読点, comma), `。` (句点, period), `：` (colon), `！` (exclamation),
  `？` (question) — fullwidth versions.
- In prose mixed with ASCII, fullwidth punctuation is standard.
- In code spans and code blocks, ASCII only (D-04).
- Headings: Japanese headings typically end without punctuation. Safe for MD026.
- Quotation marks: `「...」` (inner) or `『...』` (outer/book titles).

**Loanwords and tech nouns:**

- Japanese uses katakana for foreign tech terms: `プラグイン` (plugin), `フック` (hook),
  `テンプレート` (template), `フレームワーク` (framework), `エージェント` (agent),
  `スキル` (skill), `ワークフロー` (workflow), `コマンド` (command), `ツールキット` (toolkit).
- `Slash commands` → `スラッシュコマンド` (existing ja.md uses this pattern).
- `MCP` → keep as `MCP` (acronym, not katakana-ized in technical contexts).
- `RTK` → keep as `RTK`.
- Install: `インストール` (katakana) or `セットアップ`.
- Standalone: `スタンドアロンインストール` or `単体インストール`.
- Complement: `補完インストール` or `コンプリメントインストール`.
- Upgrading: `v3.xからのアップグレード`.

**Markdownlint gotchas for ja:**

- Same CJK width notes as zh.
- Table cells with katakana are visually double-width; GitHub renders correctly.
- MD040 — ensure ` ```bash ` and ` ```text ` on all fenced blocks.
- Japanese sentence-final `。` in prose is fine in list items — does NOT trigger MD026
  (that rule only applies to headings).
- Headings in Japanese do not end with `。` — this is a natural convention and avoids
  any MD026 edge case.

**Length factor:** Japanese is typically −5% to +5% relative to English (katakana compounds
are concise; particle-heavy prose can add lines). Net: expect ja to land close to 202 lines.

**Punctuation:**

- No space before `:` in Korean (unlike French).
- Headings: keep as noun phrases, no trailing punctuation.

**Loanwords and tech nouns:**

- Korean uses Hangul phonetic transcription for some tech terms and keeps English for others.
  Common patterns in developer docs: `플러그인` (plugin), `후크` or `hook` (hook — keep

- Standalone: `독립 설치`.

- Upgrading: `v3.x에서 업그레이드`.

**Markdownlint gotchas for ko:**

## Markdownlint Pitfalls

### Rule: MD026 — No trailing punctuation in headings

**Risk:** HIGH for French (`:` after headings is a temptation), MEDIUM for all languages.

- French specific: never end a `##` or `###` heading with ` :` even though French
  typography prescribes a colon with space before it in prose. Headings in the README are
  noun phrases — they should not end with `:`.
- Japanese/Korean specific: do not end headings with `。` or `。` fullwidth period.
- All languages: do not end headings with `?` even for "Who Is This For"-style headings.

### Rule: MD040 — Fenced code blocks must have a language

**Risk:** HIGH if a translator writes a fence without a language specifier.

Every `\`\`\`` opening must have a language immediately after: ` ```bash `, ` ```text `.
The existing translations all correctly use ` ```bash ` for shell commands and ` ```text ` for
the file-tree Structure block. New translations must follow the same pattern.

**Specific risk:** The Install Modes section introduces new fenced blocks. Ensure all three
Install Modes code blocks (Standalone, Complement, and the `--mode` override example) use
` ```bash `.

### Rule: MD031 — Blank lines around fenced code blocks

**Risk:** MEDIUM — especially when inserting the new Install Modes section.

Always have a blank line before ` ``` ` and after ` ``` `. Do not put a fenced block
immediately after a `>` blockquote or a list item with no blank line.

### Rule: MD032 — Blank lines around lists

**Risk:** LOW in this README (lists are primarily markdown tables, not bullet lists). But
the `### Standalone install` prose uses a short paragraph followed immediately by a code
fence — ensure there is a blank line between the last prose line and the ` ```bash `.

### Rule: MD024 — Duplicate headings

**Risk:** LOW (already handled by `.markdownlint.json` `siblings_only: true`).

Within a single translation file, each `##` heading must be unique within its parent `#`
and each `###` heading must be unique within its parent `##`. The README.md structure has
no duplicate headings within a section, so translations mirroring it will not trip this rule.

The cross-file case (e.g., all 8 translations have `## Killer Features`) is handled by
`siblings_only: true` — no action needed.

### Rule: MD029 — Ordered list style

**Risk:** LOW — the README.md uses `### 1.` as a heading, not a markdown ordered list.
The only actual ordered list in README.md is the Quick Start `### 1. Global Setup (once)`
which is a heading, not a list item. No ordered lists in body content.

### CJK-Specific: Table alignment

Markdownlint does NOT enforce column alignment in table separators. The ` | --- | ` separator
does not need to match the visual width of CJK cells. Write clean short separators:

```markdown
| 功能 | 描述 |
|------|------|
```

This is valid — do not try to pad with spaces to match CJK character visual widths.

### French: Space-before-colon in prose

French typography uses `Configuration :` (thin space before colon). In markdown prose,
`markdownlint-cli` does NOT flag this. However, inside a `> ` blockquote or table cell,
verify the rendered output is not confused with a list continuation. In practice, the
README's blockquotes are simple one-liners — no issue.

## Line-Count Edge Cases

### The `wc -l` definition

`wc -l` counts the number of newline characters (`\n`) in a file. A file with N lines of
text and a final trailing newline has N newlines and reports N lines. A file with no trailing
newline has N-1 newlines and reports N-1 lines.

**Verified:** All existing files (README.md and all 8 translations) end with `\n` (0x0a).
[VERIFIED: `tail -c 1 ... | xxd` shows `0a` for all 9 files]

**Implication:** New translation files MUST end with a trailing newline. If a file is written
without a trailing newline, `wc -l` returns `actual_line_count - 1`, which could push a file
that is borderline (e.g., 162 lines visually) to 161, still passing. Or push a 162-line file
(visual) to 161 — fine. But for a file near the 161 floor, the off-by-one matters.

**Rule:** Always ensure translations end with a final `\n`. The `Write` tool used during
execution must append a newline at the end of the content string.

### The Makefile integer arithmetic floor

```bash
MIN=$(( README_LINES * 80 / 100 ))
MAX=$(( README_LINES * 120 / 100 ))
```

With `README_LINES=202`:

- `202 * 80 = 16160; 16160 / 100 = 161` (exactly 161, no rounding issue)
- `202 * 120 = 24240; 24240 / 100 = 242` (exactly 242, no rounding issue)

No floor/ceiling ambiguity for this specific source length. The band is exactly 161–242
inclusive.

### Edge: German near 242

If a German translation approaches 242, the safe strategy is to merge two short consecutive
paragraphs in the Install Modes section into one paragraph. This saves 1–2 lines without
losing content. Do NOT delete the `> **Important:**` block — it is required by D-12.

### Edge: CJK near 161 (unlikely)

If a CJK translation falls below 161 (unlikely given 202-line structure), the safe strategy
is to expand an overly terse paragraph in the Killer Features or Components sections with one
or two additional explanatory sentences. The Killer Features table rows are fixed (7 rows from
English source) so they cannot be the target of expansion; target prose paragraphs instead.
