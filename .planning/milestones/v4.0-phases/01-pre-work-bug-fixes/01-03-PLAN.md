    **Pattern to apply at every mutation site:**

    Before (current shape, site 1 at lines ~201-246):

    ```bash
    if command -v python3 &>/dev/null; then
        if python3 - "$SETTINGS_JSON" "$HOOK_COMMAND" << 'PYEOF' 2>/dev/null
    import json, sys
    # ... mutation body ...
    PYEOF
        then
            echo -e "  ${GREEN}✓${NC} ..."
        else
            echo -e "  ${RED}✗${NC} Failed to configure — add manually"
        fi
    else
        echo -e "  ${YELLOW}⚠${NC} python3 not found ..."
    fi
    ```

    After (wrapped):

    ```bash
    if command -v python3 &>/dev/null; then
        # BUG-05: backup settings.json before mutation
        SETTINGS_BACKUP="${SETTINGS_JSON}.bak.$(date +%s)"
        cp "$SETTINGS_JSON" "$SETTINGS_BACKUP"

        if python3 - "$SETTINGS_JSON" "$HOOK_COMMAND" << 'PYEOF' 2>/dev/null
    import json, sys
    # ... mutation body (UNCHANGED) ...
    PYEOF
        then
            echo -e "  ${GREEN}✓${NC} ..."
        else
            cp "$SETTINGS_BACKUP" "$SETTINGS_JSON"
            echo -e "  ${RED}✗${NC} JSON merge failed — restored from backup: $SETTINGS_BACKUP"
            exit 1
        fi
    else
        echo -e "  ${YELLOW}⚠${NC} python3 not found ..."
    fi
    ```

    Key constraints:

    1. `SETTINGS_BACKUP` is assigned at the top of each python3-available branch. No global shared variable — each block recomputes `$(date +%s)` at its own point in time (D-14 timestamp format).
    2. `cp` (not `mv`) — the original file must still be in place when python3 reads it.
    3. On failure (`else` branch of the `if python3 ...; then`): `cp "$SETTINGS_BACKUP" "$SETTINGS_JSON"` BEFORE printing the error and BEFORE `exit 1`. This REPLACES the existing "Failed to configure — add manually" message, which currently swallows the failure with a zero exit.
    4. `exit 1` (not `return 1`) — D-13 requires the script terminate non-zero so callers can detect the failure.
    5. Keep the python3 heredoc body UNCHANGED. Do not refactor the merge logic.
    6. The `else` branch for `command -v python3` failure (no python3 at all) does NOT get backup/restore — nothing was mutated, nothing to restore.
    7. Preserve the success-path echo message wording from the existing code at each site (they differ between sites — do NOT copy the same message to all three).

    Apply this pattern at all three mutation sites. Each site writes its own `SETTINGS_BACKUP` assignment.

    Note on timestamp uniqueness: `$(date +%s)` has 1-second resolution. If two of the three mutation blocks execute within the same second, the second write overwrites the first backup. That is acceptable because the second mutation only happens after the first succeeds and has written its own state — the relevant recoverable state is always the most recent. Do NOT add PID suffix or nanoseconds here (Phase 4 UPDATE-05 covers collision-proof backup dir naming; this is mutation-level backup, not run-level).
