<objective>
Replace the bare `"$GEMINI_KEY"` / `"$OPENAI_KEY"` heredoc interpolation in the two config.json writers with a `python3 json.dumps()` escape pass so API keys containing `"`, `\`, `$`, or control characters produce valid JSON.

Purpose: BUG-03 is a correctness invariant. A user whose OpenAI key contains a literal backslash (as some OpenAI service-account keys do for path-like prefixes) currently produces malformed config.json that the Council cannot load — silent bricking of the feature. The fix uses `python3` (already required for Council) and matches the JSON-mutation idiom adopted by setup-security.sh (D-05/D-06).

Output: Two parallel writers patched — `scripts/setup-council.sh:178-190` (standalone Council installer) and `scripts/init-claude.sh:513-525` (inline setup within init-claude). Paired hygiene: D-08 switches the API-key `read` calls to `-rs` (silent), but that portion of the change landed with BUG-02 (already in Wave 1 plan 01-02). This plan only covers the JSON-escape portion.
</objective>

<!-- Output: ~/.claude/council/config.json with RFC 8259 JSON -->

Heredoc template (both scripts, structure is identical):

```bash
cat > "$CONFIG_FILE" << CONFIGEOF
{
  "gemini": {
    "mode": "$GEMINI_MODE",
    "api_key": "$GEMINI_KEY",
    "model": "gemini-3-pro-preview"
  },
  "openai": {
    "api_key": "$OPENAI_KEY",
    "model": "gpt-5.2"
  }
}
CONFIGEOF
```

Problem: if GEMINI_KEY = `AIza"xyz`, the output file contains `"api_key": "AIza"xyz"` → invalid JSON.

Fix shape: compute an escaped JSON-string form (including the surrounding quotes) in a shell variable, then interpolate WITHOUT adding `"..."` around it.
</interfaces>
