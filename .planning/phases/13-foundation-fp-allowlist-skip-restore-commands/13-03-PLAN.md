  <action>
Create `commands/audit-restore.md` with the following structure.

### Section 1 — Header

```markdown
# /audit-restore — Remove a False-Positive Exception from the Allowlist
```

### Section 2 — Purpose

Two-paragraph block. Paragraph 1: "Remove an entry from `.claude/rules/audit-exceptions.md` when an exception turns out to be a real bug — for example, after a refactor changes the threat model or the Council (`/council audit-review` — Phase 15) marks a previously-suppressed finding `disputed`." Paragraph 2: "Removal is guarded by an interactive `[y/N]` prompt that defaults to NO. The full entry block is displayed before the prompt so the user sees exactly what will be deleted."

### Section 3 — Usage

The `## Usage` section in `commands/audit-restore.md` MUST contain:

1. An H2 heading `## Usage`.
2. A fenced code block with language tag `text` containing exactly one line:
   - `/audit-restore <file:line> <rule>`
3. The bold prose label `**Examples:**` followed by a blank line.
4. A bullet list with these two entries:
   - `/audit-restore src/foo.ts:42 SEC-XSS`
   - `/audit-restore scripts/setup-security.sh:142 SEC-RAW-EXEC`

### Section 4 — When to Use

Bullet list:

- After a refactor changes the data flow that justified the original exception.
- After a Council `audit-review` pass (Phase 15) returns `disputed` for a previously-suppressed entry.
- When the rule itself is removed from the auditor's rule set (the entry is now meaningless).
- DO NOT use to "shut up" a stale exception you don't understand — read the original Reason field first; if the Reason no longer applies, that's a sign you genuinely want to restore.

### Section 5 — Process

#### Step 1 — Parse Arguments

```bash
set -euo pipefail

ARGS=("$@")
if [ "${#ARGS[@]}" -ne 2 ]; then
    printf 'audit-restore: usage: /audit-restore <file:line> <rule>\n' >&2
    exit 2
fi

FILE_LINE="${ARGS[0]}"
RULE="${ARGS[1]}"

PATH_PART="${FILE_LINE%:*}"
LINE_PART="${FILE_LINE##*:}"

case "$LINE_PART" in
    ''|*[!0-9]*)
        printf 'audit-restore: <file:line> must end with :<positive integer>, got %q\n' "$FILE_LINE" >&2
        exit 2
        ;;
esac

EXC_FILE=".claude/rules/audit-exceptions.md"
HEADING="### ${PATH_PART}:${LINE_PART} — ${RULE}"
```

(The em-dash in the `HEADING` string is a literal U+2014 character. Type it directly — do not escape.)

#### Step 2 — Find Entry

```bash
if [ ! -f "$EXC_FILE" ]; then
    printf 'audit-restore: %s does not exist; nothing to restore\n' "$EXC_FILE" >&2
    exit 1
fi

if ! grep -Fxq -- "$HEADING" "$EXC_FILE"; then
    printf 'audit-restore: no entry found for %s:%s:%s\n' "$PATH_PART" "$LINE_PART" "$RULE" >&2
    exit 1
fi
```

#### Step 3 — Display Entry Block

Print the heading and its body block to stdout. Use awk to extract the block from heading until next H3, H2, or EOF.

```bash
awk -v h="$HEADING" '
    BEGIN { in_block = 0 }
    $0 == h { in_block = 1; print; next }
    in_block && /^### / { exit }
    in_block && /^## /  { exit }
    in_block { print }
' "$EXC_FILE"
```

#### Step 4 — Confirm Deletion

```bash
printf '\nRemove this entry? [y/N]: '
ANSWER=""
if [ -r /dev/tty ]; then
    read -r ANSWER < /dev/tty
else
    read -r ANSWER
fi

case "$ANSWER" in
    y|Y) ;;
    *)
        printf 'Aborted. No changes.\n'
        exit 0
        ;;
esac
```

(The `< /dev/tty` fallback handles the piped-context pattern documented in CONTEXT.md "Established Patterns". If `/dev/tty` is unreadable — e.g. in CI without a TTY — fall back to plain `read` from stdin.)

#### Step 5 — Build New File (block removed)

```bash
NEW_TMP="$(mktemp)"
trap 'rm -f "$NEW_TMP"' EXIT

awk -v h="$HEADING" '
    BEGIN { skip = 0; pending_blank = 0 }
    # Buffer a blank line: if the next line is the heading, drop the blank too.
    /^$/ {
        if (skip) next
        if (pending_blank) print prev_blank
        prev_blank = $0
        pending_blank = 1
        next
    }
    $0 == h {
        skip = 1
        pending_blank = 0  # drop the blank we just buffered (the one before the heading)
        next
    }
    skip && /^### / { skip = 0 }
    skip && /^## /  { skip = 0 }
    skip { next }
    {
        if (pending_blank) { print prev_blank; pending_blank = 0 }
        print
    }
    END { if (pending_blank) print prev_blank }
' "$EXC_FILE" > "$NEW_TMP"

# Sanity check — the heading must no longer be present.
if grep -Fxq -- "$HEADING" "$NEW_TMP"; then
    printf 'audit-restore: deletion failed — heading still present in output\n' >&2
    exit 1
fi
```

The sentinel-blank logic ensures:
- A blank line that precedes the deleted heading is also removed (no double-blank residue).
- The block is delimited by the next `### ` (another entry), `## ` (another H2 — defensive: should not occur given Plan 13-01 schema), or EOF.

#### Step 6 — Atomic Replace

```bash
mv "$NEW_TMP" "$EXC_FILE"
```

#### Step 7 — Confirm to User

```bash
printf '\033[0;32m✓\033[0m Removed exception: %s:%s — %s\n' "$PATH_PART" "$LINE_PART" "$RULE"
printf '\nFile updated: %s\n' "$EXC_FILE"
printf 'Note: changes are NOT staged. Run `git add %s` and commit when ready.\n' "$EXC_FILE"
```

### Section 6 — Key Principles

Bullet list:

- **Default-N confirmation** — pressing Enter aborts. Only `y` or `Y` proceeds. (D-08)
- **Display before delete** — the user sees the full block (heading + Date/Council/Reason) before the prompt.
- **Exact triple match** — `<path>:<line>:<rule>` byte-for-byte. Same key as `/audit-skip`.
- **Block-only removal** — only the matched heading + its body bullets + the preceding blank line are deleted. Frontmatter, intro paragraph, `## Entries` H2, and other entries are untouched.
- **Atomic write** — temp + `mv`, never partial.
- **No git side effects** — write the file only; staging is the user's call.

### Section 7 — Related Commands

- `/audit-skip <file:line> <rule> <reason...>` — add an exception. Inverse of this command.
- `/audit` — runs the audit pipeline that consults `audit-exceptions.md` (Phase 14).
- `/council audit-review` — confirms or rejects findings; can flag a stale exception as `disputed` (Phase 15).

### Markdownlint compliance

- Every fenced code block has a language tag — MD040.
- Blank lines around fenced blocks and lists — MD031/MD032.
- No trailing punctuation on headings — MD026.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Slash-command argv → file deletion | The path/line/rule arguments determine what gets deleted from a Claude-auto-loaded rule file. |
| TTY input → confirmation gate | A confirmed `y` triggers irreversible (within the file) data removal. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-10 | Tampering / Unauthorized data destruction | Unconfirmed deletion | mitigate | Mandatory `[y/N]` prompt with default-N (D-08). Pressing Enter, typing `n`, typing anything other than `y`/`Y` all abort. The full entry block is printed BEFORE the prompt so the user sees exactly what will be deleted. |
| T-13-11 | Tampering | awk block-deletion takes too much (collateral damage) | mitigate | Stop conditions are explicit (`/^### /`, `/^## /`, EOF). The post-deletion sanity check (`grep -Fxq -- "$HEADING" "$NEW_TMP"` must not match) catches deletion failure. The pre-deletion `grep -Fxq` already confirmed the heading exists exactly once on its own line, so the awk loop has a deterministic stop. |
| T-13-12 | Spoofing | User passes an unrelated triple to delete a different entry | mitigate | Exact-triple match means the user cannot accidentally delete entry A by specifying triple B — they must type the exact path, line, and rule of the target. There is no fuzzy matching, no glob expansion. |
| T-13-13 | Denial of Service | Malicious rule file causes awk to loop / fail | accept | The rule file is repo-tracked under git review; an attacker who can write malicious markdown to it can already cause much worse damage (it's auto-loaded into Claude context). Risk is governed by repo write access, not by this command. |
| T-13-14 | Elevation of Privilege | Path traversal in `<path>` argument | accept | The path is used only as a string in heading-comparison and in user-facing diagnostic printf. It is NOT used as a filesystem operand (no `cat $PATH`, no `rm $PATH`). The only filesystem write is `mv "$NEW_TMP" "$EXC_FILE"` to a hardcoded path. |
</threat_model>
