<objective>
Wire `bootstrap_base_plugins()` from `scripts/lib/bootstrap.sh` (created in plan 21-01) into both `scripts/init-claude.sh` and `scripts/init-local.sh`. Each installer must:
1. Parse `--no-bootstrap` flag (and respect `TK_NO_BOOTSTRAP=1` env var)
2. Source `lib/bootstrap.sh` (curl-fetched in init-claude.sh; local-path in init-local.sh)
3. Call `bootstrap_base_plugins()` BEFORE the first mode-resolution call
4. Re-source `detect.sh` AFTER the call so `HAS_SP` / `HAS_GSD` reflect post-bootstrap reality
5. Document `--no-bootstrap` in the `--help` / usage output

Critical asymmetry between the two installers (RESEARCH.md Pitfall 1):
- `init-claude.sh` parses argparse FIRST (lines 24-51), then sources libs (63-92). Bootstrap call goes immediately after the lib-source block.
- `init-local.sh` sources libs FIRST (lines 31-38), then parses argparse (81-122). The `source bootstrap.sh` line goes in the early lib block, but the `bootstrap_base_plugins` call goes AFTER argparse + MODE validation, before re-run delegation.

Purpose: Bring the bootstrap step into both installer entry points without breaking existing v4.3 behavior.
Output: Two file edits — both installers now offer the bootstrap prompt by default and honor `--no-bootstrap`.
</objective>
